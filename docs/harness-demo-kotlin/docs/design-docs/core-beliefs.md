# Core Beliefs — 팀 핵심 원칙

이 문서는 팀의 핵심 설계 원칙을 기록합니다.
에이전트와 팀원 모두가 의사결정 시 이 원칙을 참고합니다.

## 개발 원칙

### 1. 워크트리 격리 원칙

모든 작업은 `task-start.sh`로 시작한 워크트리 내에서 수행합니다.
main / develop 브랜치를 직접 수정하는 것은 금지합니다.

**이유**: 에이전트와 사람 모두 실수할 수 있습니다. 격리는 실수의 영향 범위를 제한합니다.

### 2. 검증 우선 원칙

코드 병합 전 5가지 검증기(빌드/테스트/린트/보안/문서)를 모두 통과해야 합니다.
검증기를 우회하거나 비활성화하는 것은 예외적인 상황에만 허용합니다.

**이유**: 검증은 선택이 아닌 필수입니다. 검증 없이 병합된 코드는 팀 전체에 부담을 줍니다.

### 3. 리포지터리 = 단일 진실 원천 (Single Source of Truth)

설계 결정, 아키텍처 논의, 팀 원칙은 반드시 이 리포지터리에 문서화됩니다.
Slack/Jira/구두 합의는 에이전트에게 존재하지 않습니다.

**이유**: 에이전트는 리포지터리 밖의 정보에 접근할 수 없습니다.

### 4. 에이전트 가독성 우선

코드와 문서는 에이전트가 직접 추론할 수 있도록 작성합니다.
불명확한 약어, 내부 용어, 구두 합의에 의존하지 않습니다.

**이유**: 에이전트가 자율적으로 작업하려면 컨텍스트가 명시적이어야 합니다.

### 5. 레이어 의존성 방향

controller → service → repository → domain

역방향 참조는 금지합니다.

**이유**: 단방향 의존성은 테스트와 변경을 쉽게 만듭니다.

### 6. 하네스 진화 원칙

에이전트가 반복적으로 같은 실수를 한다면, 그것은 에이전트 문제가 아니라 하네스 문제입니다.
실패는 하네스를 개선하는 신호로 해석합니다.

**이유**: 모델을 탓하는 대신 환경을 개선하는 것이 더 효과적입니다.

## 코딩 원칙

### 명시적 > 암묵적
```kotlin
// Bad: 암묵적
fun process(data: Any) { }

// Good: 명시적
fun processUserRegistration(request: UserRegistrationRequest): UserRegistrationResult { }
```

### 경계에서 검증
시스템 경계(HTTP 요청 수신, 외부 API 응답 파싱)에서만 데이터를 검증합니다.
내부 함수 간에는 신뢰하고 전달합니다.

### 테스트는 명세서
테스트는 "어떻게 동작하는가"를 명세하는 문서입니다.
테스트명이 곧 기능 명세가 되도록 작성합니다.

## 금지 사항

- `// TODO: fix later` — 지금 고치거나 Jira 티켓 생성
- 빈 catch 블록 — 최소한 로그라도 남기기
- 하드코딩된 환경값 — 환경 변수 또는 설정 파일 사용
- 테스트 없는 비즈니스 로직 추가
