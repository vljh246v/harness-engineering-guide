## 1. 하네스 엔지니어링이란?

```
Agent = Model + Harness
```

하네스(Harness)는 AI 모델을 뺀 나머지 전부를 가리킵니다. 프롬프트, 도구, 스킬, 로그, 피드백 루프까지 — 에이전트가 일하는 환경 그 자체입니다.

마구(馬具)가 말의 힘을 올바른 방향으로 이끌듯이, 하네스는 AI 에이전트가 멋대로 날뛰지 않고 안전하게 동작하도록 잡아주는 제어 구조 전체를 말합니다.

> "초기 진행이 예상보다 더뎠는데, Codex의 역량이 부족해서가 아니라 **환경이 제대로 갖춰지지 않았기 때문**이었다."  
> — OpenAI Engineering

같은 모델인데 하네스만 갈아끼웠더니 52.8%에서 66.5%로 성능이 올라간 사례가 있습니다. 모델을 바꾸는 것보다 환경을 바꾸는 게 더 효과적일 수 있다는 뜻입니다.

## 2. 왜 하네스가 필요한가: 3가지 근본 문제

원칙이나 도구를 설명하기 전에, AI 코딩 에이전트가 사람과 근본적으로 다른 지점을 짚어야 합니다. 이건 다음 모델 업데이트로 고쳐질 버그가 아닙니다. LLM 기반 에이전트의 **구조적 속성**이고, 하네스가 존재하는 이유입니다.

### 세션 간 기억 소실

모든 세션은 백지에서 시작합니다. 에이전트는 어제 대화를 기억하지 못합니다. 어제의 실수도, 어제의 결정도, 어제 20분 동안 설명해준 DTO 패턴도 — 세션이 끊기는 순간 전부 사라집니다.

월요일에 "이 프로젝트는 response DTO를 record로 만들어야 해"라고 설명합니다. 에이전트가 알겠다고 하고 잘 따릅니다. 화요일, 새 세션 — 또 설명합니다. 수요일, 또 설명합니다. 매일이 Groundhog Day입니다.

더 심각한 건 실수의 기억도 날아간다는 겁니다. 어제 `.env` 파일을 커밋해서 30분 동안 고쳤는데, 오늘 또 `.env`를 커밋합니다. 어제의 "이러면 안 된다"가 오늘에는 존재하지 않으니까요.

**왜 구조적인가?** LLM은 stateless function입니다. 입력을 받아서 출력을 내놓고 끝. 호출 사이에 상태가 남지 않습니다. 모델이 아무리 똑똑해져도, 세션 경계는 기억 경계입니다.

**하네스의 대응**: 에이전트가 기억 못 하면, 환경이 대신 기억하면 됩니다.

- `golden-principles.md` — 반복된 실수를 영속적 규칙으로 변환. 에이전트가 기억 못 해도 파일이 알려줌
- `agent-failures.md` — 실패의 원본 기록. 날짜, 증상, 원인이 남아있어서 세션이 바뀌어도 이력 추적 가능
- `failure-patterns.md` + Feedforward — 작업 시작 시 최근 실패 통계를 자동으로 보여줌. "기억"이 아니라 "읽기"로 해결
- 리포지터리 자체가 기록 시스템 — 파일에 적혀있지 않으면 에이전트에게 존재하지 않음. 대화에서 합의한 건 파일로 남겨야 살아남음

### 컨텍스트 불안 (Context Anxiety)

에이전트의 컨텍스트 창은 유한합니다. 프로젝트에 golden principles 50개, 플레이북 10개, 아키텍처 문서, 소스 파일 200개가 있는데 컨텍스트에는 20%밖에 안 들어갑니다. 뭘 넣고 뭘 뺄 것인가?

이게 **컨텍스트 불안**입니다. 넣어야 할 것 같은 정보는 많은데, 전부 넣으면 정작 중요한 게 소음에 묻힙니다. 적게 넣으면 에이전트가 핵심 규칙을 모른 채 코드를 짭니다. 어떻게 해도 불안합니다.

"컨텍스트 창이 커지면 해결되지 않나?"라고 생각할 수 있는데, 창이 커지면 코드베이스도, 규칙도, 문서도 같이 커집니다. 비율의 문제라서 영원히 풀리지 않습니다. 그리고 설령 다 들어가더라도, 규칙이 100개면 규칙이 없는 것과 같습니다. 정보가 많을수록 주의력은 분산됩니다.

**왜 구조적인가?** 유한한 창에 무한히 늘어나는 컨텍스트를 담아야 하는 건 아키텍처적 제약입니다. 1M 토큰 창이 와도 대규모 모노레포의 ADR, principles, 문서를 전부 담으면 주의력 희석으로 오히려 성능이 떨어집니다.

**하네스의 대응**: 전부 넣는 대신, 필요한 것만 필요한 시점에 넣으면 됩니다.

- CLAUDE.md를 목차로 — 60~100줄. 백과사전이 아니라 지도
- 스킬로 분리 — `/harness-task` 같은 명령으로 호출할 때만 로드
- 스코프 CLAUDE.md — `src/controller/CLAUDE.md`는 controller에서 작업할 때만 읽힘
- 서브에이전트 — 컨텍스트 방화벽. 코드 리뷰를 서브에이전트에 맡기면 리뷰 과정이 메인 컨텍스트를 오염시키지 않음
- 점진적 공개 — 에이전트가 지금 필요한 정보만, 필요한 시점에

### 자기 평가 불능 (Self-Evaluation Failure)

에이전트에게 기능을 구현하라고 시키면, 끝나고 "완료했습니다! 테스트도 작성했습니다"라고 말합니다. 확인해보면 기능은 절반만 됐고, 테스트는 happy path 하나뿐입니다. 에이전트가 거짓말을 하는 게 아닙니다 — 진짜로 다 됐다고 생각합니다.

더 명확한 실험이 있습니다. 에이전트에게 자기가 방금 짠 코드를 리뷰하라고 시키면 "문제없습니다"가 나옵니다. 같은 코드를 다른 에이전트에게 리뷰 시키면 이슈가 5개 나옵니다. 자기가 만든 걸 자기가 평가하면 객관성이 사라집니다.

**왜 구조적인가?** LLM은 다음에 올 가능성이 가장 높은 토큰을 생성합니다. 코드를 생성한 뒤에 가장 가능성 높은 토큰은 "잘 했다"는 긍정 완료 문장이지, "여기 버그가 있다"는 자기 비판이 아닙니다. 생성과 평가는 본질적으로 다른 프로세스인데, 같은 추론 체인 안에서 둘 다 하라는 건 구조적으로 불가능합니다.

**하네스의 대응**: 만든 놈이 판단하지 못하면, 다른 놈이 판단하면 됩니다.

- Planner → Generator → Evaluator 패턴 — 구현한 에이전트와 평가하는 에이전트를 분리
- 검증기 파이프라인 (`task-finish.sh`) — 빌드·테스트·린트·보안·문서, 에이전트의 "완료" 주장과 무관하게 객관적 검증
- `verify-task.sh` — 구현 중에도 수시로 검증 가능. 끝났다고 느끼기 전에 확인
- 훅 — 위험한 동작을 에이전트가 인지하기 전에 실시간으로 잡음

### 3가지 문제는 서로를 강화한다

이 문제들은 독립적이지 않습니다. 어제 세션에서 컨텍스트를 어떻게 구성해야 효과적이었는지 기억 못 하니까, 오늘도 같은 컨텍스트 불안을 겪습니다. 컨텍스트가 제한돼서 자기 평가 기준을 충분히 로드 못 하니까, 평가가 더 부정확해집니다. 자기 평가를 못 하니까, 무엇을 잊었는지도 모릅니다.

```
문제                      하네스의 대응
─────────────────────    ──────────────────────────────
세션 간 기억 소실         → 파일 기반 영속적 기억 (golden-principles, agent-failures, failure-patterns)
컨텍스트 불안             → 점진적 공개 + 스코프 분리 (CLAUDE.md, 스킬, 서브에이전트)
자기 평가 불능            → 독립 검증 (Evaluator, 검증기, 훅)
```

하네스 엔지니어링은 이 세 가지 구조적 한계에 대한 체계적 대응입니다. 이어지는 원칙과 메커니즘은 전부 여기서 출발합니다.

## 3. 근본 원칙: 이렇게 생각해야 한다

위 3가지 근본 문제를 전제로, 하네스를 설계할 때 지켜야 하는 원칙들입니다.

### 에이전트를 신뢰하지 마라

앞에서 본 것처럼 에이전트는 기억하지 못하고, 맥락이 제한되고, 자기 평가를 못 합니다. 이 셋을 전제로 깔아야 나머지가 다 이해됩니다. 검증기가 왜 필요한지, Evaluator가 왜 독립적이어야 하는지, golden-principles가 왜 쌓여야 하는지 — 전부 "에이전트는 혼자 두면 실수한다"에서 출발합니다.

### 컨텍스트는 희소 자원이다

이 제약에서 하네스의 거의 모든 설계 결정이 나옵니다:

- CLAUDE.md를 100줄 이하로 → 정말 중요한 것만 남기고 나머지는 링크
- 스킬로 분리 → 필요할 때만 로드
- 서브에이전트 → 메인 컨텍스트를 오염시키지 않고 작업 위임
- 스코프 CLAUDE.md → 해당 디렉토리에서 작업할 때만 관련 규칙 로드
- 점진적 공개 → 에이전트가 지금 필요한 정보만, 필요한 시점에

"전부 다 알려주면 되지 않나?"는 통하지 않습니다. 정보가 100개면 아무것도 중요하지 않은 것과 같습니다.

### 에이전트의 범위를 좁혀라

에이전트가 건드릴 수 있는 범위가 넓을수록 하네스를 촘촘하게 짜기 어렵습니다. Martin Fowler는 이걸 "서비스 토폴로지 제한"이라고 불렀습니다.

한 에이전트에게 "전체 시스템을 리팩터링해줘"라고 하면 하네스가 감당할 수 없습니다. 대신 "이 서비스의 이 레이어에서 이 기능을 고쳐줘"로 좁히면 검증기가 제대로 잡아줄 수 있습니다. 워크트리 격리, exec-plan, escalation-policy 전부 이 원칙의 실현입니다.

### 문서는 무시된다, 코드로 강제하라

"controller에서 repository를 직접 주입하지 마세요"를 ARCHITECTURE.md에 써놓으면 에이전트가 읽을 수도 있고 안 읽을 수도 있습니다. 읽었어도 무시할 수 있습니다.

같은 규칙을 ArchUnit 테스트로 만들면 물리적으로 불가능합니다. 검증기가 차단하니까요.

```
문서에 적음         → "읽어주면 좋겠다" (무시 가능)
린트 규칙으로 만듦   → "경고는 뜨지만 우회 가능"
검증기에 추가        → "통과 못 하면 병합 자체가 안 됨"
pre-commit hook     → "실행 자체가 불가능"
```

처음부터 전부 코드로 강제할 필요는 없습니다. 문서로 시작하되, 반복되면 위쪽으로 올리는 겁니다. 그게 재발 방지 루프의 인코딩 우선순위입니다.

### 사람의 역할은 코딩이 아니라 환경 설계

OpenAI 팀은 5개월 동안 코드를 한 줄도 직접 작성하지 않고 약 100만 라인짜리 제품을 출시했습니다. 엔지니어 1인당 하루 평균 3.5개의 PR을 처리했고, 팀이 3명에서 7명으로 늘어나는 동안 처리량은 오히려 증가했습니다.

그러면 사람은 매일 뭘 하는가? 코드를 짜는 대신 이런 걸 합니다:

- CLAUDE.md를 다듬고, 너무 길어지면 docs/로 분리
- agent-failures.md를 검토하고, 반복 패턴을 golden-principles로 승격
- escalation-policy에 새 카테고리를 추가
- 검증기가 못 잡는 케이스를 발견하면 새 검증기를 추가
- exec-plan을 리뷰하고, 고위험 작업에 승인

코드를 짜는 시간이 줄고, 에이전트가 잘 일할 수 있는 환경을 설계하는 시간이 늘어납니다. 이게 가능했던 건 AGENTS.md, 워크트리 격리, 자동 검증 파이프라인 같은 하네스를 잘 깔아뒀기 때문입니다.

## 4. 핵심 원리: Feedforward vs Feedback

하네스의 제어 장치는 크게 두 방향으로 나뉩니다.

```
┌─────────────────────────────────────────────────────────────┐
│                    하네스 제어 구조                           │
│                                                             │
│  Feedforward (사전 제어)       Feedback (사후 수정)          │
│  ─────────────────────        ─────────────────────         │
│  행동하기 전에                 행동한 후에                     │
│  방향을 잡아줌                 틀린 걸 바로잡음                │
│                                                             │
│  • CLAUDE.md                  • 훅 (Hooks)                  │
│  • 스킬 (Skills)              • 검증기 (Validators)          │
│  • golden-principles.md       • 평가기 에이전트              │
│  • escalation-policy.md       • agent-failures.md           │
│  • failure-patterns.md        • 린터 (Linter)               │
│  • 플레이북 (Playbooks)                                     │
└─────────────────────────────────────────────────────────────┘
```

### Feedforward — 미리 알려주기

에이전트가 뭔가를 하기 전에 방향부터 잡아주는 장치들입니다.

- **CLAUDE.md**: 이 프로젝트에서 어떻게 일해야 하는지 알려주는 지도
- **golden-principles.md**: 전에 이런 실수를 했으니까 이번엔 하지 마, 라는 누적 규칙
- **escalation-policy.md**: DB 스키마나 인증 로직 같은 건 혼자 건드리지 말고 사람한테 먼저 물어봐, 라는 경계선
- **failure-patterns.md**: 최근 30일간 어떤 검증기가 자주 실패했는지 통계. task-start.sh가 알아서 보여줌
- **플레이북**: DB 마이그레이션처럼 반복되는 고위험 작업을 단계별 레시피로 만들어둔 것

### Feedback — 틀리면 바로 잡기

에이전트가 이미 뭔가를 한 다음에 자동으로 수정하게 만드는 장치들입니다.

- **훅**: Edit/Write 같은 도구 실행 전후에 자동으로 끼어드는 스크립트
- **검증기**: 빌드, 테스트, 린트, 보안 검사를 통과 못 하면 병합 자체가 안 됨
- **평가기 에이전트**: 구현한 에이전트 말고 다른 에이전트가 독립적으로 검증
- **agent-failures.md**: 실수가 터지면 바로 기록. 이 기록이 쌓여서 golden-principles로 승격됨

> 에이전트는 자기가 한 일을 과하게 칭찬하는 경향이 있습니다. "다 했습니다!" 했는데 실제로는 반만 된 경우가 흔합니다. 그래서 **직접 만든 놈 말고 다른 놈이 검증**해야 합니다.

## 5. 5가지 레버

### 왜 하필 5개인가?

5가지 레버는 이론적으로 나눈 게 아닙니다. Claude Code가 실제로 제공하는 제어 수단이 딱 5개이고 (gpters.org 분석), 각각이 서로 다른 문제를 해결합니다.

| 이걸 빼면 | 무슨 일이 생기는가 | 그래서 필요한 레버 |
|-----------|--------------------|--------------------|
| 규칙을 안 알려주면 | 프로젝트 컨벤션 무시하고 자기 방식대로 짬 | **시스템 프롬프트** |
| 전문 지식을 통째로 넣으면 | 컨텍스트 낭비. 안 넣으면 지식 부족으로 실수 | **스킬 + 플레이북** |
| 외부 시스템을 못 보면 | Jira 티켓이나 DB 스키마를 매번 사람한테 물어봄 | **MCP 서버** |
| 작업을 분할 못 하면 | 긴 컨텍스트에서 앞부분 까먹고 뒷부분에서 실수 | **서브에이전트** |
| 실시간 통제가 없으면 | force push, rm -rf 같은 걸 사후에야 발견 | **훅** |

하나라도 빠지면 하네스에 구멍이 납니다. 다섯 가지가 서로 다른 구멍을 막고 있는 셈이죠.

### 레버 1: 시스템 프롬프트 (CLAUDE.md)

에이전트가 프로젝트에 처음 들어와서 제일 먼저 읽는 파일입니다. 일종의 지도 역할을 합니다.

중요한 건 **목차(Map)로 쓰되 백과사전으로 쓰지 않는 것**입니다. 60~100줄이면 충분합니다. 상세한 규칙은 `docs/` 아래 문서에 두고, CLAUDE.md에서는 링크만 걸어둡니다.

```markdown
# CLAUDE.md 예시

## 작업 유형별 진입점
| 작업 | 먼저 읽을 문서 |
|------|---------------|
| 새 기능 | ARCHITECTURE.md → docs/golden-principles.md |
| 버그 수정 | logs/trends/failure-patterns.md → docs/agent-failures.md |
| 고위험 변경 | docs/escalation-policy.md → 사람 승인 필수 |

## 워크플로우
task-start.sh → 구현 → verify-task.sh → task-finish.sh

## 금지 사항
→ docs/golden-principles.md 참고
```

이렇게 표 하나면 에이전트가 "어떤 작업이든 뭘 먼저 읽어야 하는지" 바로 알 수 있습니다.

#### 점진적 공개 (Progressive Disclosure)

모든 규칙을 CLAUDE.md 하나에 몰아넣으면 세 가지 문제가 생깁니다:

1. 컨텍스트 창은 유한한데, 거대한 파일이 정작 중요한 정보를 밀어냄
2. 지침이 100개면 지침이 아님 — 그냥 소음
3. 시간이 지나면 낡은 규칙들의 무덤이 됨

대신 **스코프 CLAUDE.md**를 씁니다. 디렉토리마다 그 레이어에 해당하는 규약만 적어두는 겁니다.

```
src/controller/CLAUDE.md    # "service만 호출, repository 직접 접근 금지"
src/service/CLAUDE.md       # "비즈니스 로직은 여기에만"
src/repository/CLAUDE.md    # "외부 API 호출 금지"
```

에이전트가 `src/controller/` 안에서 작업하면 거기 있는 CLAUDE.md만 읽게 됩니다. 필요한 맥락만, 필요한 시점에.

### 레버 2: 스킬 (Skills) & 플레이북 (Playbooks)

**스킬**은 에이전트가 평소에는 안 읽다가 특정 작업을 할 때만 꺼내 쓰는 전문 지식입니다. `/harness-task`처럼 슬래시 명령으로 호출하면 그때 로드됩니다.

```
.claude/skills/
├── harness-task/SKILL.md    # 작업 시작 → 구현 → 검증 → 병합 전체 자동화
├── code-review/SKILL.md     # PR 리뷰 체크리스트
└── debugging/SKILL.md       # 체계적 버그 추적 절차
```

**플레이북**은 조금 다릅니다. 스킬이 "에이전트의 능력"이라면, 플레이북은 "작업의 레시피"입니다. DB 마이그레이션이나 API 엔드포인트 추가처럼 반복되는 고위험 작업을 단계별 체크리스트로 만들어둔 겁니다.

중요한 건 각 체크리스트 항목이 실제 실패에서 나왔다는 점입니다. "이거 빼먹어서 장애 난 적 있으니까 꼭 확인해"가 한 줄 한 줄 쌓인 결과물이죠.

### 레버 3: MCP 서버

에이전트가 코드만 보는 게 아니라 Jira 티켓을 조회하고, DB 스키마를 확인하고, Slack으로 알림을 보낼 수 있어야 합니다. MCP 서버가 이 다리 역할을 합니다.

이게 없으면 에이전트가 "이 티켓 내용이 뭐예요?" 하고 사람한테 매번 물어봐야 합니다.

### 레버 4: 서브에이전트 (Sub-agents)

메인 에이전트가 코드 리뷰를 하면서 동시에 구현도 해야 한다고 생각해보세요. 컨텍스트 창이 터집니다.

서브에이전트는 **컨텍스트 방화벽** 역할을 합니다. 코드 리뷰를 서브에이전트에 맡기면, 리뷰 과정에서 쌓이는 정보가 메인 에이전트의 컨텍스트를 오염시키지 않습니다. 결과만 간결하게 돌려받으면 됩니다.

### 레버 5: 훅 (Hooks)

에이전트가 파일을 수정하거나 명령어를 실행할 때 자동으로 끼어드는 스크립트입니다.

```json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash|Edit|Write",
      "hooks": [{ "type": "command", "command": ".claude/hooks/pre-edit-check.sh" }]
    }]
  }
}
```

이걸로 할 수 있는 것들:

- 워크트리 밖에서 소스 코드 수정하려고 하면 → 경고
- `git push --force` 시도하면 → 차단
- 파일 수정 후 자동으로 린트 돌리기

사람이 옆에서 지켜보지 않아도 위험한 동작을 실시간으로 잡아줍니다.

## 6. 멀티에이전트 패턴

### Planner → Generator → Evaluator

```
사람이 요청
 ▼
Planner     "로그인 기능 추가해줘" → 상세 사양으로 확장, exec-plan에 저장
 ▼
Generator   exec-plan 보고 워크트리에서 구현
 ▼
Evaluator   Generator가 만든 코드를 독립적으로 검증
 ▼
task-finish.sh  검증기 5개 통과해야 병합
```

핵심은 **Generator가 자기 코드를 자기가 평가하지 않는다**는 겁니다. "다 됐습니다"라고 말해놓고 실은 반만 된 경우가 너무 흔하거든요.

### GC (Garbage Collection) 에이전트

주기적으로 돌면서 리포지터리를 건강하게 유지합니다:

1. 검증기 실패 로그를 분석해서 `failure-patterns.md` 업데이트
2. 오래된 워크트리 정리
3. 문서가 코드와 어긋나는 부분(드리프트) 감지
4. `agent-failures.md`를 훑어보고 3회 이상 반복된 실패가 있으면 구조적 강제로 승격 제안

사람이 안 챙겨도 리포지터리가 서서히 썩는 걸 막아주는 역할입니다.

## 7. 워크트리 기반 강제 검증

### 왜 워크트리인가?

| 상황 | 워크트리 없이 | 워크트리 있으면 |
|------|-------------|---------------|
| 에이전트가 실수함 | develop이 바로 오염됨 | 격리된 브랜치에서만 영향 |
| 여러 작업 동시 진행 | 충돌 위험 | 독립 워크트리로 병렬 처리 |
| 검증 강제 | 건너뛸 수 있음 | task-finish.sh 안 거치면 병합 자체가 안 됨 |
| 실패 후 디버깅 | 상태 복원 어려움 | 워크트리 보존돼서 원인 분석 가능 |

### 전체 플로우

```
task-start.sh <task-id>
  ├─ git worktree 생성 (.worktrees/<task-id>/)
  ├─ exec-plan 스켈레톤 자동 생성 (docs/exec-plans/active/<task-id>.md)
  ├─ 세션 로그 초기화 (logs/sessions/<task-id>/session.jsonl)
  └─ failure-patterns.md 미리보기 출력

      [여기서 코드 작성]

verify-task.sh <task-id>              ← 구현 중에 수시로 돌려봄. 병합은 안 함
  └─ 검증기 5개 실행, 결과만 보여줌

task-finish.sh <task-id>              ← 최종 관문
  ├─ 01-build → 02-test → 03-lint → 04-security → 05-docs
  │  하나라도 실패하면 BLOCKED
  └─ 전부 통과하면
     develop 동기화 → merge --no-ff → 워크트리 삭제
```

`verify-task.sh`를 따로 뺀 이유가 있습니다. `task-finish.sh`는 검증이 통과하면 바로 병합까지 해버리니까, 아직 작업 중인데 "지금 상태 괜찮나?" 확인만 하고 싶을 때 쓸 수가 없거든요. 그래서 검증만 따로 돌릴 수 있는 스크립트를 분리한 겁니다.

### exec-plan 자동 생성

`task-start.sh`를 실행하면 워크트리만 만드는 게 아니라, `docs/exec-plans/active/<task-id>.md` 스켈레톤도 같이 만들어줍니다.

```markdown
---
task_id: PROJ-101
type: feat
status: in-progress
created: 2026-04-15T09:00:00Z
---

# PROJ-101

## 배경 및 목표
[작업 목표를 작성하세요]

## 구현 단계
- [ ] 1. 영향 범위 파악
- [ ] 2. 구현
- [ ] 3. 테스트 작성
- [ ] 4. verify-task.sh로 검증
- [ ] 5. task-finish.sh로 병합
```

빈 플랜이라도 파일이 존재하면 채우게 됩니다. 아예 없으면 "나중에 쓰지 뭐" 하다가 계획 없이 코딩부터 시작하거든요.

### 언어에 종속되지 않는 구조

`harness.config.sh` 파일 하나만 프로젝트에 맞게 바꾸면 됩니다. 검증기 스크립트는 그대로 쓸 수 있습니다.

```bash
# Kotlin          → BUILD_CMD="./gradlew compileKotlin"
# TypeScript      → BUILD_CMD="npm run build"
# Python          → BUILD_CMD="pip install -r requirements.txt -q"
```

### 에이전트가 읽을 수 있는 오류 메시지

일반 린터는 `ERROR: Import not allowed` 한 줄만 뱉습니다. 에이전트가 이걸 보고 뭘 어떻게 고쳐야 하는지 모릅니다.

하네스의 검증기는 다릅니다:

```
[BUILD FAILURE] validators/01-build.sh
  커맨드: ./gradlew compileKotlin
  종료 코드: 1

  오류:
  src/main/kotlin/UserService.kt:45: Unresolved reference: userId

  수정 방법:
  1. UserService.kt:45에서 userId 변수 선언을 확인하세요
  2. 함수 파라미터나 클래스 필드로 정의되어 있는지 확인하세요
  참고: ARCHITECTURE.md
```

파일명, 라인 번호, 수정 방향까지 다 알려줍니다. 에이전트가 이 메시지만 읽고 스스로 고칠 수 있어야 합니다.

## 8. 관측 가능성

> **에이전트에게 보이지 않으면 존재하지 않는다**

로그를 남기더라도 에이전트가 직접 읽을 수 없는 형태면 의미가 없습니다.

### 세션 로그 (로컬, append-only)

```jsonl
{"ts":"...","event":"task_start","task":"PROJ-101"}
{"ts":"...","event":"exec_plan_created","path":"docs/exec-plans/active/PROJ-101.md"}
{"ts":"...","event":"tool_use","tool":"Bash","cmd":"./gradlew build","exit":0}
{"ts":"...","event":"validator","name":"02-test","result":"fail","error":"NullPointerException"}
{"ts":"...","event":"validator","name":"02-test","result":"pass"}
{"ts":"...","event":"task_finish","result":"success","validators_passed":5}
```

에이전트의 모든 행동이 타임스탬프와 함께 쌓입니다. 이건 개인 작업 기록이라 git에는 올리지 않습니다.

### 검증기 이력 (git 추적, 팀 공유)

어떤 태스크에서 어떤 검증기가 언제 성공/실패했는지 기록합니다. 이건 팀 전체가 볼 수 있도록 git에 올립니다.

### 실패 패턴 (GC 에이전트가 분석)

검증기 이력을 분석해서 "최근 30일간 가장 자주 실패한 패턴"을 정리합니다. `task-start.sh`가 작업 시작할 때 이걸 자동으로 보여줘서, 에이전트가 같은 실수를 또 하지 않도록 미리 경고합니다. 이게 Feedforward입니다.

## 9. 리포지터리를 기록 시스템으로

### 문서 구조

```
CLAUDE.md (~60줄, 목차)                    ← 에이전트 진입점
│
├── ARCHITECTURE.md                        ← 레이어 규칙
│
├── docs/
│   ├── design-docs/
│   │   ├── index.md                       ← ADR 목록 + 템플릿
│   │   ├── core-beliefs.md                ← 팀 핵심 원칙
│   │   └── ADR-XXXX-*.md                  ← 구조적 결정들
│   │
│   ├── golden-principles.md               ← 실패에서 승격된 규칙
│   ├── agent-failures.md                  ← 에이전트 실패 로그
│   ├── escalation-policy.md               ← 사람 승인 필수 변경
│   ├── playbooks/                         ← 반복 작업 레시피
│   │
│   ├── exec-plans/
│   │   ├── active/                        ← 진행 중 계획
│   │   ├── completed/                     ← 완료된 계획 아카이브
│   │   └── tech-debt-tracker.md           ← 기술 부채 추적
│   │
│   └── QUALITY_SCORE.md                   ← 도메인별 품질 현황
│
└── logs/
    ├── sessions/                          ← 세션 로그 (로컬)
    ├── validators/history.jsonl           ← 검증 이력 (git 추적)
    └── trends/failure-patterns.md         ← 실패 패턴 (git 추적)
```

### ADR (Architecture Decision Records)

"왜 이렇게 하기로 했는지"를 기록하는 문서입니다. 보통 ADR은 Context/Decision/Consequences 세 섹션으로 끝나는데, 하네스에서는 **Enforcement 섹션**을 하나 더 넣습니다.

```markdown
## Enforcement ← 이게 핵심
이 결정은 어떻게 강제되는가?
- 훅: pre-edit-check.sh — 워크트리 밖 수정 시 경고
- 검증기: task-finish.sh — 검증 없이 병합 불가
- golden-principles: GP-001
```

결정을 문서에만 써놓으면 무시됩니다. 린트 규칙이든, 검증기든, 훅이든 — 기계적으로 강제될 때 비로소 의미가 있습니다.

### 기술 부채 추적

"이건 나중에 고치자"고 넘어간 건 `docs/exec-plans/tech-debt-tracker.md`에 적어둡니다. 눈에 안 보이면 잊히니까요. 해결하면 해당 행을 지우고 커밋 메시지에 `refs: TD-001` 같이 남깁니다.

### 왜 꼭 리포지터리에?

Slack에서 합의한 내용, Google Docs에 정리한 아키텍처 — 이런 건 에이전트한테 존재하지 않습니다. 에이전트가 접근할 수 있는 건 리포지터리 안에 있는 파일뿐입니다.

그리고 규칙을 하나의 거대한 파일에 다 몰아넣는 것도 안 됩니다. 컨텍스트 창은 유한한 자원인데 쓸모없는 옛날 규칙이 정작 중요한 정보를 밀어내거든요.

## 10. 하네스 진화 전략: 재발 방지 루프

### 핵심: 하네스는 완성품이 아니라 진화하는 시스템이다

에이전트가 같은 실수를 반복하면, 그건 에이전트 탓이 아니라 **하네스가 부족한 겁니다**. 에이전트를 더 똑똑하게 만들 수는 없지만, 환경을 더 촘촘하게 만들 수는 있으니까요.

### 재발 방지 루프

```
에이전트가 실수함
  │
  ▼
agent-failures.md에 기록
  │  증상 / 근본 원인 / 대응 / 도메인
  │
  ├─ 같은 패턴이 2번째 → golden-principles.md에 원칙으로 승격
  │
  └─ 같은 패턴이 3번째 → 린트/검증기/훅으로 구조적 차단
                          (문서가 아니라 코드로 막음)
```

구체적으로 어떻게 돌아가는지, .env 파일 사고를 예로 들어보겠습니다.

**1회차**: 에이전트가 `git add -f .env`로 시크릿이 담긴 파일을 커밋함.  
→ `agent-failures.md`에 기록만 합니다.

**2회차**: 또 같은 일이 일어남.  
→ `golden-principles.md`에 원칙 승격:
```
GP-002: .env 파일을 git에 포함하지 않는다
- 왜: git history에 한 번 들어가면 완전 제거가 어려움
- 대신: .gitignore + 환경 변수로 관리
- 강제: 04-security.sh에서 git ls-files로 체크
```

**3회차**: 세 번째라면 문서로는 안 된다는 뜻입니다.  
→ pre-commit hook에서 `.env` 파일 `git add` 자체를 차단. 에이전트가 원하더라도 커밋이 물리적으로 안 됩니다.

이게 "재발 방지 루프"입니다. 실패 → 기록 → 원칙 → 구조적 강제. 하네스가 스스로 단단해지는 메커니즘이죠.

### 에스컬레이션 정책: 에이전트의 경계

모든 작업을 에이전트한테 맡길 수 있는 건 아닙니다. `escalation-policy.md`에는 에이전트가 혼자 건드리면 안 되는 영역을 정해둡니다:

- DB 스키마 변경
- 인증/인가 로직
- 결제/과금 로직
- 보안 설정
- 하네스 자체 변경 (검증기 비활성화 같은 것)

이런 변경은 exec-plan에 영향 범위, 롤백 방법, 검증 방법을 적고 사람 승인을 받아야 합니다.

### 강제 수단의 세기

같은 규칙이라도 어떻게 강제하느냐에 따라 효과가 다릅니다.

```
문서화 (golden-principles, ADR)     ← 가장 약함. 무시 가능
린터 규칙                           ← 경고는 뜨지만 우회 가능
검증기 (task-finish.sh)             ← 강력. 통과 못 하면 병합 자체가 안 됨
훅 + pre-commit                     ← 가장 강력. 실행 자체가 불가능
```

처음에는 문서로 시작하되, 반복되면 위쪽으로 올려야 합니다.

하네스는 완성품이 아닙니다. 작은 것부터 시작해서, 실패가 쌓이면 한 단계씩 키워나가면 됩니다. 실패가 하네스를 성장시키는 연료니까요.

## 11. 하네스는 복사해서 끝이 아니다

템플릿을 복사하면 스크립트나 검증기는 바로 동작합니다. 하지만 하네스의 절반은 **팀이 직접 채워야 하는 문서**입니다. 빈 채로 두면 에이전트가 참고할 게 없어서 자기 판단으로 움직이게 됩니다.

### 프로젝트 시작 시 반드시 채워야 하는 것

**escalation-policy.md** — 에이전트가 실수한 다음에 추가하면 이미 늦습니다. 시작 전에 "혼자 건드리면 안 되는 영역"을 정해둬야 합니다.

**harness.config.sh의 Git Flow / PR 규칙** — 빌드 커맨드뿐 아니라 브랜치 네이밍, 커밋 메시지 형식, PR 제목 규칙, 기본 리뷰어까지 정하지 않으면 에이전트가 자기 맘대로 합니다.

**ARCHITECTURE.md** — 레이어 구조가 없으면 에이전트가 controller에서 repository를 직접 호출하는 코드를 만들 수 있습니다.

**core-beliefs.md** — 팀의 핵심 원칙 3~5개. "왜 이렇게 하는가"의 근거가 됩니다.

### 처음에는 비워둬도 되는 것

**golden-principles.md** — 앉아서 미리 쓰는 게 아닙니다. 에이전트가 실수하면 `agent-failures.md`에 기록하고, 같은 패턴이 반복되면 여기로 승격됩니다. 시간이 지나면 "우리 프로젝트에서 자주 하는 실수" 모음이 자연스럽게 쌓입니다. 단, 이미 알고 있는 위험 패턴이 있다면 시드로 넣어두는 게 좋습니다.

**playbooks/** — 같은 유형의 작업에서 비슷한 실수가 3번 반복되면 그때 레시피를 만듭니다.

**agent-failures.md** — `/harness-task`가 3회 실패하면 자동으로 기록하고, 하네스가 못 잡은 문제를 사람이 발견하면 수동으로 추가합니다.

### 핵심: 설정과 문서가 안 채워지면 하네스가 아니라 그냥 스크립트 모음이다

검증기가 5개 돌아가고 워크트리가 격리돼도, escalation-policy가 비어있으면 에이전트가 DB 스키마를 맘대로 바꿉니다. golden-principles가 없으면 같은 실수를 3번째 해도 아무런 승격이 일어나지 않습니다. PR 규칙이 없으면 커밋 메시지가 제각각입니다.

하네스의 진짜 힘은 스크립트가 아니라, 그 스크립트가 읽는 **문서와 설정**에 있습니다.

> 구체적인 설정 예시와 작성 가이드는 `harness-engineering-example/README.md`를 참고하세요.

## 12. 어디서부터 시작하면 되는가

한 번에 전부 갖출 필요 없습니다. 작은 것부터 시작해서, 실패가 쌓이면 하네스를 키워나가면 됩니다.

### 1단계: 에이전트에게 규칙을 알려주기

에이전트가 프로젝트에 들어왔을 때 "여기서 어떻게 일해야 하는지"를 알 수 있으면 됩니다. 시스템 프롬프트 하나와, 위험한 동작을 잡아주는 훅 하나면 시작할 수 있습니다.

### 2단계: 작업을 격리하기

에이전트가 기본 브랜치를 직접 건드리지 못하게 막고, 필요한 지식을 필요한 시점에만 주는 구조를 만듭니다. 워크트리로 격리하고, 스킬로 지식을 분리합니다.

### 3단계: 병합 전에 검증을 강제하기

검증을 "하면 좋은 것"이 아니라 "안 하면 병합이 안 되는 것"으로 만듭니다. 빌드, 테스트, 린트, 보안, 문서 — 이 중 하나라도 실패하면 코드가 들어올 수 없어야 합니다. 구현 중에 수시로 검증할 수 있는 방법도 있어야 합니다.

### 4단계: 결정을 기록하고, 규칙을 기계적으로 강제하기

"왜 이렇게 하기로 했는지"를 기록하고, 그 결정이 문서에만 머물지 않고 린트/테스트/훅으로 강제되는 구조를 만듭니다. 반복 작업에는 레시피를 만들어 에이전트가 따라가게 합니다.

### 5단계: 하네스가 스스로 진화하게 만들기

실패를 기록하고, 반복되면 원칙으로 승격하고, 계속 반복되면 코드로 막는 루프를 돌립니다. 에이전트가 혼자 건드리면 안 되는 경계를 정하고, 주기적으로 하네스 자체의 건강을 점검합니다.

> 구체적인 파일별 체크리스트는 `harness-engineering-example/README.md`의 "적용 방법"을 참고하세요.
