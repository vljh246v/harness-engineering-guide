# Golden Principles — 에이전트가 지켜야 할 누적 규칙

`agent-failures.md`에서 반복된 실패가 승격된 원칙 목록입니다.
각 원칙은 실제 실패 사례에서 유래하며, 가능한 한 구조적으로 강제됩니다.

## 승격 기준

- `agent-failures.md`에서 같은 패턴 2회 이상 → 이 문서에 원칙 추가
- 같은 패턴 3회 이상 → 린트/테스트/검증기로 자동 강제 필수

---

## 원칙 목록

### GP-001: 워크트리 없이 소스 코드를 수정하지 않는다

- **왜**: 워크트리 밖 수정은 develop/main을 직접 오염시킴
- **대신**: `./scripts/task-start.sh <task-id>` 실행 후 워크트리에서 작업
- **강제**: `.claude/hooks/pre-tool-use.sh` — src/ 파일 수정 시 워크트리 여부 확인

### GP-002: .env / 시크릿 파일을 git에 포함하지 않는다

- **왜**: git history에 한 번이라도 들어가면 완전 제거가 어려움
- **대신**: `.gitignore`에 등록 + 환경 변수 또는 시크릿 관리 도구 사용
- **강제**: `04-security.sh` — `git ls-files`로 커밋된 .env 감지

### GP-003: 테스트 없이 비즈니스 로직을 추가하지 않는다

- **왜**: 테스트 없는 코드는 다음 변경에서 조용히 깨짐
- **대신**: 기능 코드와 테스트를 같은 커밋에 포함
- **강제**: `02-test.sh` — 테스트 전부 통과 필수

### GP-004: 검증기를 우회하지 않는다

- **왜**: 검증기 우회로 병합된 코드는 팀 전체에 부담
- **대신**: 검증기가 실패하면 코드를 수정. 검증기가 잘못됐으면 검증기를 고치고 기록
- **강제**: `task-finish.sh` — 검증 통과 없이 병합 불가

> 새 원칙 추가 시 `GP-{N+1}` 번호를 부여하고, 반드시 **왜/대신/강제** 세 항목을 채우세요.
> **강제** 항목이 "문서만"이면 3회 재발 시 구조적 강제로 격상해야 합니다.
